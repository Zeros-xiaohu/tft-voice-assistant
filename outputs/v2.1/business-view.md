 ## 1. 总业务逻辑

 ### 1.1 业务是什么

 > **帮云顶玩家在 30 秒内做出更好的阵容决策。**

 这不是数据查询工具，也不是聊天 bot。它是一个**缩短决策时间、提升决策质量的辅助决策系统**。

 ### 1.2 核心业务指标

| 指标 | 含义 | 为什么重要 |
|------|------|-----------|
| 决策时间 | 从"有疑问"到"知道该怎么做" | 游戏里只有 30 秒 |
| 决策质量 | 采纳建议后的实际对局结果 | 没用就没人用 |
| 采纳率 | 用户采纳建议的比例 | 信任度和可操作性 |
| 使用频率 | 单局交互轮次 | 融入度 |

 ### 1.3 业务价值链路

 云顶玩家的一局游戏天然切成两个阶段，产品也对应两个模式：

 ```
 一局云顶之弈 (约 35 分钟)
 │
 ├── 战前 (1-3 分钟) ──────── 对战模式
 │   排队/加载中
 │   用户时间充裕，心态放松
 │   行为: 研究版本、了解阵容、确定方向
 │   目标: 建���版本认知
 │
 ├── 战中 (30 分钟) ──────── 吃鸡模式
 │   对局进行中
 │   用户时间紧迫，需要快速决策
 │   行为: 告知局面、获取建议、执行操作
 │   目标: 提升实时决策质量
 │
 └── 战后 ────────────── (v2.3 复盘)
 ```

 两个模式不是独立功能，**这是覆盖一局游戏完整决策链路的两个业务环节**。

 ### 1.4 为什么分成两个模式

| | 合并成一个模式 | 分成两个模式 |
|---|---|---|
| 用户状态 | 混杂，"我正在打"和"我在研究"用同一个界面 | 清晰，"排队时点左边，打游戏时点右边" |
| Agent 行为 | 不知道用户是在战前还是战中，建议节奏不对 | 对战模式不记状态，吃鸡模式记状态 |
| 输出方式 | 统一输出，战中场景信息过载 | 对战给完整卡片，吃鸡给结构化行动建议 |
| 数据策略 | 全部实时查 | 对战调 MetaTFT，吃鸡 Agent 自主决定调不调 |

 **根本原因: 战前和战中是两种完全不同的用户心态和决策需求，不能用同一个产品形态覆盖。**

 ## 2. 对战模式 — 业务逻辑

 ### 2.1 业务定位

 > **帮用户在开打前建立版本认知。**

 用户不赶时间，有耐心看完整信息。需求是"这个版本什么强、怎么玩、我想玩什么"。

 ### 2.2 用户画像和使用场景

| 场景 | 用户行为 | 心态 |
|------|---------|------|
| 排位排队中 | 打开产品，查强势阵容 | "这把玩什么能上分" |
| 刚更新版本 | 查新版本改了啥、什么阵容起来了 | "版本变了，我该学什么" |
| 和朋友讨论 | 查某个英雄配装、某个阵容细节 | "金克斯到底带羊刀还是无尽" |
| 打完一把复盘 | 查刚才那把的阵容是不是版本答案 | "我刚才玩的阵容到底强不强" |

 ### 2.3 业务逻辑链

 这是一个**从泛到精的漏斗模型**:

 ```
 用户: "当前版本什么阵容强"
   → 返回 Top 3 阵容卡片（阵容名、胜率、核心英雄、装备、海克斯）
     │
     ├── 用户: "幻灵战队具体怎么玩"
     │   → 返回阵容详情（每个英雄的装备推荐、站位建议）
     │
     ├── 用户: "金克斯带什么装备"
     │   → 返回英雄配装卡片（Top 5 装备 + 胜率数据）
     │
     ├── 用户: "反曲弓和大剑能合什么"
     │   → 返回装备合成卡片（合成公式 + 推荐顺序）
     │
     └── 用户: 决定了，就玩幻灵战队
         → [转入吃鸡模式追踪此阵容]
 ```

 每一步都在缩小范围: 版本 → 阵容 → 英雄 → 装备。

 ### 2.4 四种查询类型的业务关系

 | 查询类型 | 用户想知道 | 输入 | 输出 | 属于漏斗哪层 |
 |---------|-----------|------|------|------------|
 | 阵容推荐 (comp_tier) | "这版本什么强" | 自然语言/流派名 | Top 3 阵容 | **顶层: 建立概览** |
 | 英雄配装 (hero_build) | "这个英雄怎么配" | 英雄名 | 5 件装备 + 胜率 | **中层: 深入英雄** |
 | 装备合成 (item_recipe) | "这两个能合什么" | 2 个散件 | 合成列表 | **底层: 执行细节** |
 | 装备排行 (item_tier) | "什么装备厉害" | 无 | Top 12 装备 | **辅助: 横向对比** |

 ### 2.5 "一键转入"的业务意义

 ```
 对战模式查到阵容 → 点"开一局试试" → 吃鸡模式自动初始化新对局 + 预填目标阵容
 ```

 这不是一个功能按钮。这是**用户决策链路的闭环**: 研究 → 决定 → 开打，全程在同一个产品里完成。

 ## 3. 吃鸡模式 — 业务逻辑

 ### 3.1 业务定位

 > **帮用户在对局中做出实时最优决策。**

 用户时间紧迫，每轮只有 30 秒。需求不是"了解"，而是"告诉我现在该怎么做"。

 ### 3.2 用户画像和使用场景

| 场景 | 回合 | 用户行为 | 心态 |
|------|------|---------|------|
| 开局决策 | 2-1 | 告知装备+海克斯选项，选海克斯 | "开局了，该走什么方向" |
| 方向锁定 | 3-2 | 告知来牌+装备，定主C阵容 | "牌来了，该玩什么" |
| 中期运营 | 3-5 ~ 4-2 | 告知经济+血量，问D牌还是存钱 | "要不要D牌，要不要升人口" |
| 危机应对 | 4-1+ | 告知核心未成型/血量低 | "打不过了，怎么办" |
| 决赛圈 | 5-1+ | 告知剩余对手 | "剩3家，怎么站位" |

 ### 3.3 业务逻辑链: 感知-决策-反馈循环

 ```
 用户告知局面 (装备/海克斯/来牌/回合/偏好)
   │
   ▼
 Agent 感知: 理解用户当前处境
   │  "3-2，物理装备，战士之心，有德莱文，想玩赌狗"
   │  → 局面: 3-2正常节奏，物理开局，适合赌狗路线
   │
   ▼
 Agent 规划: 分析需要什么数据
   │  "需要知道: 什么赌狗阵容适配物理装备+战士之心？
   │             手上的散件能合什么装备？"
   │
   ▼
 Agent 执行: 调工具获取数据
   │  search_comps(物理, 赌狗) → 德莱文赌狗/诺手赌狗
   │  search_item_recipes(反曲弓, 大剑) → 巨人杀手/羊刀/...
   │
   ▼
 Agent 决策: 综合数据 + 游戏知识 → 建议
   │  "推荐德莱文赌狗。理由: 装备完美适配，战士之心强化前排。
   │   现在合成巨人杀手给德莱文。保持50金慢D追3星。"
   │
   ▼
 Agent 追问: 缺什么就问
   │  "你现在多少金币？商店刷到了什么？"
   │
   ▼
 用户执行/回应 → 下一轮循环
 ```

 ### 3.4 Agent 的自主性: 为什么不是"用户问一句 AI 答一句"

| | 被动查询 (v1.0/v2.0) | Agent (v2.1) |
|---|---|---|
| 谁决定查什么 | 用户说"查布里茨" | **Agent 自己判断需要什么数据** |
| 查多少 | 一次一个 | **Agent 可能并行查阵容+装备合成+英雄配装** |
| 要不要追问 | 不问，答完结束 | **缺关键信息时主动追问** |
| 记不记得 | 不记得 | **记住整局状态，下一轮不用重说** |

 Agent 的自主性 = **从"你问我答"变成"我帮你想该问什么"**。

 ### 3.5 用户偏好的业务意义

 "赌狗"、"运营"、"上9"这些不是技术概念，是**云顶玩家的游戏策略分类**:

| 偏好 | 用户特征 | Agent 策略 |
|------|---------|-----------|
| 赌狗 | 喜欢低费主C、追3星、早期发力 | 只推 1-2 费主C阵容，强调 Lv5-Lv6 慢D节奏 |
| 运营 | 喜欢稳扎稳打，正常节奏 | 推 4 费主C阵容，强调经济管理 |
| 上9 | 喜欢存钱速升，玩高费阵容 | 强调不D牌只存钱，速升 9 级找 5 费卡 |

 Agent 不理解这一点 = 给赌狗玩家推高费阵容 = 建议完全不可用。

 ### 3.6 Agent 的决策框架（业务语言版）

| 步骤 | 技术名 | 业务含义 |
|------|--------|---------|
| 感知 | State Parser | "听懂用户在说什么" |
| 规划 | Decision Engine Plan | "想清楚需要什么数据才能帮用户做决策" |
| 执行 | Tool Executor | "拿到实时数据了" |
| 合成 | Synthesize | "数据 + 游戏知识 → 告诉用户该怎么做" |
| 记忆 | Session Store | "记住了刚才发生的事，下次接着来" |

 ## 4. 业务视角

 ### 4.1 用户视角: 我是怎么用的

 #### 对战模式

 ```
 排位排队 → 打开产品 → 点对战模式
   → 说话: "当前版本法师阵容"
   → 看到 Top 3 阵容: 幻灵、诅咒战士、重装枪手
   → 点幻灵战队看详情: 核心英雄、装备、海克斯
   → 查金克斯配装: 羊刀+无尽+轻语
   → 决定玩这个 → 点"开一局试试"
 ```

 #### 吃鸡模式

 ```
 游戏开始 → 手机放旁边 → 打开吃鸡模式
   → 说: "开始新对局，想玩赌狗"
   → Agent: "你开局拿到什么装备？"
   → 说: "反曲弓大剑"
   → Agent: [查数据] → "推荐德莱文赌狗。合成巨人杀手。商店有德莱文就买。"
   → 2-1: "海克斯有战士之心和法师之心"
   → Agent: "选战士之心。完美匹配。"
   → 3-2: "德莱文差2张2星"
   → Agent: "保持50金慢D，别急。第二个海克斯告诉我。"
   → ...持续循环...
   → 打完 → "结束对局" → "第2名"
   → Agent: "已记录。第2名不错！"
 ```

 用户在使用中**不是"查询"心态，是"被辅助"心态**。Agent 像坐在旁边的朋友。

 ### 4.2 竞品视角

| | MetaTFT | TFT-consider | tft-helper | tft-ai-coach | 我们 |
|---|---|---|---|---|---|
| 输入 | 🖱️ 打字 | 📸 截图 | 🖱️ 浏览 | 📸 截图 | 🎤 **语音** |
| 安装 | 浏览器 | 桌面应用 | 浏览器 | 桌面应用 | 浏览器 |
| 战中可用 | ❌ | ✅ | ❌ | ✅ | ✅ |
| 战前可用 | ✅ | ❌ | ✅ | ❌ | ✅ |
| 决策辅助 | 数据展示 | AI分析 | 数据展示 | AI分析 | **AI分析+偏好理解** |
| 用户关系 | "我去查" | "它帮我看" | "我翻翻" | "它帮我看" | **"我跟它一起打"** |

 核心壁垒: **语音交互 + 战前战中全覆盖 + 零安装**。这三条加在一起，竞品没有同时做到的。

 ### 4.3 价值视角

| 角色 | 对战模式提供 | 吃鸡模式提供 |
|------|------------|------------|
| 休闲玩家 | 不用研究版本 | 跟着建议走，玩得开心 |
| 排位玩家 | 快速了解版本Meta | 决策效率提升，更容易上分 |
| 新手 | 通过查英雄/装备学习游戏 | Agent 追问帮助理解游戏机制 |
| 老手 | 省去信息收集时间 | 专注策略博弈，Agent 处理信息 |

 ## 5. 业务边界

 ### 5.1 做什么

 **对战模式**
- 阵容推荐、英雄配装、装备合成、装备排行
- 从版本概览到执行细节的漏斗式查询

 **吃鸡模式**
- 感知用户局面，理解用户偏好
- 自主调用工具获取实时数据
- 综合推理给出具体操作建议
- 记住对局状态，主动追问

 **两个模式的连接**
- 对战查到的阵容可一键带入吃鸡模式追踪

 ### 5.2 不做什么

| 不做的 | 原因 |
|--------|------|
| 自动截图/屏幕感知 | Web/PWA 定位决定。语音交互是主动选择 |
| 自动操作游戏 | 安全边界。Agent 是顾问，不是代打 |
| 实时对手分析 | 用户获取对手信息的成本太高 |
| 赛后深度复盘 | v2.3 再做 |
| 用户系统/登录 | 不需要，降低使用门槛 |
| 偏好自动学习 | v2.3 再做。目前用户主动表达偏好 |
| 多版本数据 | 只支持当前版本 |
| 社交/社区 | 不满足用户核心需求 |

 ### 5.3 用户不做什么

| 用户不需要 | 产品替代了 |
|-----------|-----------|
| 打字 | 说话 |
| 安装软件 | 浏览器打开 |
| 研究版本 | 对战模式告诉你 |
| 记装备合成表 | 说两个散件就行 |
| 自己分析数据 | 吃鸡模式分析完给建议 |
| 每次都从头说 | Agent 记住了 |

 ### 5.4 用户与 Agent 的分工

 ```
 用户负责:
   · 动手操作游戏 (买棋子、D牌、升人口、合装备)
   · 告知局面 (装备、海克斯、来牌、回合)
   · 做最终决策 (采纳或忽略建议)

 Agent 负责:
   · 查数据 (MetaTFT 实时 + CDragon 本地)
   · 分析局面 (结合游戏知识推理)
   · 给建议 (具体到棋子名/装备名/回合)
   · 记状态 (Session 追踪)

 互不越界:
   · Agent 永远不替用户操作
   · 用户永远不需要自己查数据
 ```

 ### 5.5 Agent 行为约束

| 约束 | 说明 |
|------|------|
| 不确定时说"不确定" | 不编造数据。数据不足时降级用游戏知识，标注"未获取实时数据" |
| 建议可被忽略 | 不追问"你为什么不听我的" |
| 不假设用户水平 | 新手老手收到同样的建议质量 |
| 不评判用户决策 | 用户选了"错"的海克斯，基于新情况调整，不批评 |
| 每轮对话独立可理解 | session 丢失也能给出有意义建议 |

 ## 附录: 两个模式的业务差异一览

| | 对战模式 | 吃鸡模式 |
|---|---|---|
| **用户状态** | 不在游戏中，时间充裕 | 正在对局，30秒决策窗口 |
| **用户需求** | "这版本什么强" | "现在该怎么做" |
| **心态** | 研究、探索 | 紧迫、决策 |
| **产品形态** | Workflow（一问一答） | Agent（多轮记忆+自主决策） |
| **AI 行为** | 被动解析意图 | 主动规划、调用工具、追问 |
| **数据** | 每次调 MetaTFT | Agent 自主决定调不调 |
| **状态** | 不记 | 记全局 |
| **输出** | 信息卡片 | 行动建议卡片 |
| **终点** | 用户决定玩什么阵容 | 对局结束，记录排名 |
| **业务指标** | 查询准确度 | 建议采纳率、决策质量 |
